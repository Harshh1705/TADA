## What this is

**tada** — the second of two tools for the Together Fund technical intern
take-home. Given a pitch deck (PDF) and optionally a GitHub/docs link, it
extracts the deck's technical AI claims and produces a claim-by-claim verdict:
is this a fine-tune or a foundation model, is the "AI moat" architecturally
real or a thin prompt wrapper, what's actually novel vs. table-stakes — rather
than generic market/competition summarization.

Sibling tool (already built, separate repo): **FolioPy** — the portfolio
monitoring agent. `core/` here is a copy of FolioPy's `core/`, not a fresh
build. Don't re-invent config/llm/trace/research — port them over and move on.

## Command surface

```
tada config set --provider groq --api-key <key>     # same config file as FolioPy
tada config show
tada run <path/to/deck.pdf>
tada run <path/to/deck.pdf> --repo <github_url>       # optional extra grounding source
```

No `add`/`list`/`import` — every `run` is a fresh, independent analysis. Don't
force this tool into FolioPy's stateful shape; it doesn't need a watchlist.

## MVP scope (build this first, in this order, nothing else)

1. Parse deck → plain text per slide/section (PyMuPDF).
2. Extract claims → structured list, **3-4 claim categories only**:
   `model_architecture` (fine-tune vs. foundation vs. from-scratch),
   `data_moat` (proprietary data claims), `technical_differentiation`
   (what's claimed as novel), `other` (catch-all, low priority). Do not
   build a large taxonomy — 3-4 categories checked well beats 10 checked
   shallowly, and it's what actually gets narrated in the demo video.
3. Ground each claim → 1-2 searches per claim via `core/research.py`
   (already built, reused as-is).
4. Verdict per claim → `supported` / `contradicted` / `inconclusive`, with
   evidence citations, generated by one LLM call over the claim + gathered
   evidence.
5. Markdown report → `data/reports/<deck-name>-diligence.md`.

## Explicitly deferred (do NOT start these before the MVP above works end-to-end)

- **Supabase persistence of past reports.** Nice for a future "have we seen
  this claim pattern before" feature (noted as future work in FolioPy's docs
  too), but not required — a one-shot tool with no persistence still
  satisfies "structured output, multi-step reasoning, retrieval" from the
  brief. Only add `schema.sql` + a `diligence_reports` table if the MVP is
  done with meaningful time left.
- **`core/splash.py` logo swap.** Purely cosmetic. Last thing you touch, if at all.
- **`--repo <github_url>` grounding.** Useful, but the PDF-only path must work
  perfectly before adding a second input source. Stub the flag, wire it up
  only after `run <deck.pdf>` alone is solid.
- **Large claim taxonomy / sub-claims / confidence calibration beyond
  supported/contradicted/inconclusive.** Resist the urge to make the verdict
  schema fancier — it doesn't move the grading needle and it's the easiest
  place to lose an hour polishing something nobody will ask about.

## Core design decisions (and why)

- **CLI-only, no UI.** Same reasoning as FolioPy — terminal trace *is* the
  demo's "visible reasoning" requirement, and it's already free since
  `core/trace.py` is reused verbatim.
- **PyMuPDF for deck parsing**, not a general PDF-to-text library — matches
  your existing TheDocProject parsing approach, and PyMuPDF handles slide
  layouts (text blocks, positions) better than plain text extraction.
- **One-shot, not stateful.** No Supabase dependency required for the MVP —
  this tool can run with just `LLM_PROVIDER`/`LLM_API_KEY` set, which is
  actually a nice contrast to point out in the writeup: not every tool in an
  agentic toolkit needs a database.
- **Reused `core/research.py` for grounding**, not a new retrieval system —
  the "ground a claim" operation is structurally identical to "research a
  portfolio company": search, fetch, extract. Building a second research
  pipeline here would be wasted time.
- **Verdict schema kept to 3 states** (`supported`/`contradicted`/
  `inconclusive`) rather than a numeric confidence score — simpler to reason
  about, faster to build, and honest about what an LLM-grounded verdict can
  actually claim (a partner still makes the judgment call).

## Data flow (one `tada run <deck.pdf>` call)

1. `deck_parser.py`: PDF → list of (slide_number, text) via PyMuPDF.
2. `claim_extractor.py`: one LLM call over full deck text → `list[Claim]`
   (each tagged with category + originating slide number).
3. For each `Claim`: `grounding.py` → `core/research.py.search()` (1-2
   queries) + `fetch()` on top results → `list[Evidence]`.
4. `verdict.py`: one LLM call per claim, given the claim + its evidence →
   `Verdict` (supported/contradicted/inconclusive + reasoning + citations).
5. `report.py`: all verdicts → markdown file in `data/reports/`.

Every step above logs through the reused `Tracer`.

## Known open questions / assumptions to flag in the writeup

- Claim taxonomy is deliberately narrow (3-4 categories) — named as a scoping
  choice under time constraints, not an oversight.
- Verdicts are judgment-assistive, not judgment-replacing — the tool should
  never output something that reads like a go/no-go recommendation; it
  surfaces evidence for a partner to weigh.
- No persistence in the MVP — each run is stateless. Explicitly fine for this
  submission; noted as a natural extension in "future work," not a gap to
  apologize for.
- Grounding quality depends on public discoverability of technical claims —
  same coverage-ceiling caveat as FolioPy's monitoring tool.
